# Using docker-compose

While the **Dockerfile** defines the experiment environment, the **docker-compose.yml** file defines **how the experiment is executed**.

It specifies aspects such as:

- which image will be used
- which environment variables are defined
- which directories are shared with the container
- whether the experiment uses a GPU

This file completely describes how the experiment should be executed.

---

## Example docker-compose

A typical `docker-compose.yml` file for an experiment might look like this:

```yaml
services:

  experiment:

    build: .

    runtime: nvidia
    deploy: 
      resources: 
        reservations: 
          devices: 
            - driver: nvidia 
              device_ids: ['0'] 
              capabilities: [gpu]

    environment:
      - EPOCHS=50
      - LEARNING_RATE=0.001

    volumes:
      - ./data:/data:ro
      - ./results:/results
```

This file defines the container that will execute the experiment.

---

## Building the image

The `build` option indicates that Docker must build the image using the Dockerfile in the current directory.

```yaml
build: .
```

This means that before running the experiment, Docker will build the image defined by the Dockerfile.

---

## Using the GPU

To allow the container to use the server's GPU, the following configuration is used:

```yaml
runtime: nvidia
deploy: 
  resources: 
    reservations: 
      devices: 
        - driver: nvidia 
          device_ids: ['0'] 
          capabilities: [gpu] 
```

This allows the container to access CUDA and the available GPU.

---

## Environment variables

Environment variables allow experiment parameters to be defined.

```yaml
environment:
  - EPOCHS=50
  - LEARNING_RATE=0.001
```

These variables will be available inside the container and the program can read them to configure its behavior.

This allows the experiment configuration to be changed **without modifying the code or rebuilding the Docker image**.

---

## Volumes

Volumes allow directories to be shared between the server and the container.

```yaml
volumes:
  - ./data:/data:ro
  - ./results:/results
```

In this example:

- `./data` contains the input data for the experiment
- `./results` is used to store the results

Note that the data volume definition includes the `:ro` tag at the end. This means the volume is mounted in **read-only mode**. The container can see the data, open it, and operate on it, but it cannot write to that directory. This protects the data from being accidentally modified.

In the case of the results directory, this configuration is not included. By default, if nothing is specified, the directory allows both reading and writing.

Inside the container these directories will be available as:

```plain
/data
/results
```

This allows the results generated by the container to remain on the server.
